# 战斗系统修改与排错流程规范

> 目标：为 `src/wangyou`
> 战斗系统相关脚本、界面、配置、文档改动建立统一执行标准，保证每次修改都可追踪、可复现、可验证、可回档。

---

## 1. 文档目的与适用范围

本文只适用于 `src/wangyou` 下与战斗系统有关的改动，包括但不限于：

- 战斗脚本：`src/wangyou/脚本/战斗系统`
- 测试页与状态栏 UI：`src/wangyou/界面`
- 初始化配置、世界书变量、技能定义、掉落配置
- 战斗相关说明文档、开发计划、Git 流程文档

凡是涉及以下任一项，均必须按本文流程执行，不得跳步：

- 检定公式
- 伤害、治疗、护盾、状态结算
- 目标判定与命令校验
- 回合阶段推进与自动流转
- 长期字段与运行态桥接
- 测试页展示与调试信息
- 战斗相关文档更新

若修改完成后未补齐本文要求的记录、验证、文档同步与回档点，视为流程未完成。

---

## 2. 每次修改前必须执行的准备步骤

开始修改前，必须先完成以下准备，并在任务记录中写明结果：

1. 阅读本次改动直接相关的说明文档，至少覆盖：
   - `src/wangyou/DND通用战斗系统具体脚本设计.md`
   - `src/wangyou/第一版战斗系统字段边界说明.md`
   - `src/wangyou/战斗系统检定与伤害计算说明.md`
   - `src/wangyou/第一版战斗系统后续开发计划.md`
   - 涉及 Git 流程时再读 `src/wangyou/GitHub上传与回档操作指南.md`
2. 明确本次修改的目标、非目标、影响范围。
3. 判断本次改动是否涉及以下高风险类别：
   - 公式
   - 默认值
   - 目标判定
   - 阶段流转
   - 长期字段
   - 运行态桥接层
   - UI 调试信息
4. 定位改动入口文件，至少明确：
   - 主逻辑入口文件
   - 受影响的展示层文件
   - 需要同步的说明文档
5. 若当前问题可复现，必须先记录复现条件，再开始改代码。

禁止事项：

- 未读相关文档直接改公式或状态机。
- 未明确影响范围直接跨文件大面积重构。
- 未记录原问题现象就直接修改默认值或兜底逻辑。

## 3. 修改任务登记模板

每次任务开始前，必须先建立一份任务登记。可写在任务说明、提交说明或临时记录中，但字段不能缺失。

同时新增硬性归档要求：

- 只要属于中大型修改，必须在 `src/wangyou/xiugai` 下新建一份记录文件。
- 文件名必须使用“当前时间戳 + 简短主题”的形式，例如：`1775485092762-playful-rocket.md`。
- 该文件必须完整记录本次修改目标、范围、过程、行为变化、验证、文档同步、遗留问题。
- 若本次改动跨多个文件、涉及公式/默认值/长期字段/UI/验证中的任意两类及以上，默认按“中大型修改”处理，不得跳过。
- 后续排错、回档、继续开发时，优先阅读 `src/wangyou/xiugai` 中最新相关记录。

```md
## 修改任务登记

- 修改目标：
- 问题现象：
- 预期结果：
- 非目标：
- 涉及文件：
- 影响范围：
- 风险点：
- 复现方式：
- 验证方式：
```

硬性要求：

- `修改目标` 必须是可验证的结果，不得只写“优化一下”。
- `问题现象` 必须描述当前错误行为，不得只写“有 bug”。
- `涉及文件` 至少要列出计划修改文件与计划同步文档。
- `风险点` 必须写明是否可能影响公式、阶段流转、目标判定、长期字段。
- 中大型修改必须同步生成 `src/wangyou/xiugai/<时间戳>-<主题>.md` 记录文件，不能只把信息留在聊天或临时说明里。

## 4. 变更过程记录要求

修改过程中，每改一类逻辑，都必须记录以下内容：

- 为什么改
- 改了什么
- 替代方案为什么不用
- 影响到了哪些行为或文件

以下场景必须额外记录“旧行为 -> 新行为”对照，不能省略：

- 公式变更
- 默认值变更
- 目标判定变更
- 阶段流转变更
- 自动推进逻辑变更
- 结算回写变更

记录格式建议：

```md
### 行为变更记录

- 变更点：
- 原因：
- 旧行为：
- 新行为：
- 为什么不用其他方案：
- 影响范围：
```

若 UI 仅做展示增强，也必须记录：

- 新增展示项是什么
- 该展示项用于解决什么定位问题
- 是否依赖新的 payload、派生字段或日志结构

未记录修改原因和行为变化的改动，不得视为完成。

补充硬性要求：

- 若存在 `src/wangyou/xiugai/*.md` 任务记录文件，则本节内容必须写入该文件，而不是只停留在口头说明。
- 记录必须足够详细，至少能让后续维护者在不翻完整聊天记录的情况下理解“为什么改、改了哪里、怎么验证、还有什么没做完”。

## 5. 文件级变更登记要求

每个被修改的文件，都必须标记其角色。至少从下列分类中选择一项：

- 引擎：如 `engine.ts`
- 命令校验：如 `commands.ts`
- 检定与效果计算：如 `roll.ts`、`effects.ts`
- 运行态派生或桥接：如 `derive.ts`、`bridge.ts`、`store.ts`、适配层文件
- 测试页或状态栏 UI：如 `App.vue`、面板组件、测试页入口
- 配置：如 `initvar.yaml`、技能表、掉落表
- 文档：开发计划、字段边界、检定说明、Git 规范

文件级登记至少要写明：

- 文件路径
- 文件角色
- 本次改动目的
- 是否引入行为变化
- 是否需要同步其他文件

建议记录格式：

```md
- 文件：src/wangyou/脚本/战斗系统/commands.ts
  - 角色：命令校验
  - 目的：修正道具默认目标类型兜底
  - 行为变化：有
  - 需同步：战斗系统检定与伤害计算说明.md
```

## 6. 行为变化记录要求

以下行为变化属于高优先级记录对象，修改后必须写入变更说明：

- 防御结算时机
- 药剂目标选择与默认目标类型
- 提交行动后的自动推进逻辑
- 技能显示方式与展示字段
- 检定日志结构
- 伤害拆解与护盾吸收路径
- 战斗结束后的长期数据回写
- **CTB计数器初始值、重置公式、行动顺序逻辑**
- **多单位目标判定与AoE命中范围**
- **被动效果触发时机与效果类型**



1. 写明触发条件。
2. 写明旧结果与新结果。
3. 写明用户可观察差异。
4. 写明是否影响已有文档、测试页或归档结构。

示例：

```md
- 变更项：药剂目标类型缺省值
  - 触发条件：道具未配置 `目标类型`
  - 旧行为：命令校验可能返回“没有合法目标”
  - 新行为：按 `self` 兜底选择自身目标
  - 可观察差异：治疗药剂可在测试页直接正常提交
  - 需同步文档：字段边界说明、检定与伤害计算说明
```

## 7. 测试与复现记录要求

每次修改至少保留两条记录：

1. 一条“如何复现原问题”
2. 一条“如何验证修复成功”

若涉及数值、判定或状态，必须记录具体样例，至少包含：

- 使用角色或单位
- 使用技能或道具
- 关键属性或资源值
- 预期日志或结算结果

推荐最小记录模板：

```md
### 复现记录

- 场景：
- 前置条件：
- 操作步骤：
- 原问题结果：

### 验证记录

- 场景：
- 前置条件：
- 操作步骤：
- 预期结果：
- 实际结果：
```

涉及 UI 展示的改动，还必须补充：

- 具体查看位置
- 新展示项名称
- 该项是否能支撑定位问题

若无法提供复现路径或验证结果，该次修改不得提交。

## 8. 文档同步规则

战斗系统修改完成后，必须检查是否需要同步以下文档：

1. 修改公式、判定、伤害、护盾、状态结算时：
   - 必须检查 `src/wangyou/战斗系统检定与伤害计算说明.md`
2. 修改阶段性目标、已完成事项、后续优先级时：
   - 必须更新 `src/wangyou/第一版战斗系统后续开发计划.md`
3. 修改长期字段、运行态边界、命令白名单时：
   - 必须检查 `src/wangyou/第一版战斗系统字段边界说明.md`
4. 修改 Git 提交流程、回档约束、分支策略时：
   - 必须更新 `src/wangyou/GitHub上传与回档操作指南.md`
5. 只要属于中大型修改：
   - 必须新增或更新 `src/wangyou/xiugai` 下对应时间戳记录文件

硬性规则：

- 修改完如果不更新对应说明文档，视为流程未完成。
- 若判断“无需更新文档”，必须明确写出理由，不能默认跳过。
- 若属于中大型修改但未在 `src/wangyou/xiugai` 留下详细记录，视为流程未完成。

## 9. Git 工作流要求

战斗系统改动必须遵守以下 Git 约束：

1. 从功能分支进行，不直接在 `main` 堆积未隔离改动。
2. 小步提交，尽量做到 1 个逻辑点对应 1 次提交。
3. 提交说明必须写清楚“为什么改”，不能只有“update”或“fix bug”。
4. 提交前必须确认只包含本次任务相关文件。
5. 必须提供可回档点：
   - 未推送时至少给出本地最新 commit id
   - 已推送时优先使用 `git revert` 作为回退方案

禁止项：

- `git reset --hard` 作为常规回档手段
- `git push --force` 覆盖历史
- 将多个无关逻辑混成一个提交
- 未验证就直接提交

推荐最小提交顺序：

1. 文档阅读与任务登记
2. 代码或文档修改
3. 构建 / 手动验证 / 记录结果
4. 提交并保留回档点

具体 Git 命令约束以 `src/wangyou/GitHub上传与回档操作指南.md` 为准；若两者不一致，先更新 Git 指南，再执行新流程。

## 10. 给 AI 的固定执行清单

后续 AI 每次修改战斗系统时，必须按以下顺序执行，不得跳过：

1. 先读文档
2. 再定位代码
3. 再登记任务目标、现象、影响范围、风险点
4. 再实施修改
5. 再验证复现路径与修复结果
6. 再总结变化
7. 再给 commit / 回档点
8. 再确认 `src/wangyou/xiugai` 已新增本次详细记录文件（若属于中大型修改）

输出结果时，至少应包含：

- 修改目标
- 实际改动文件
- 关键行为变化
- 验证结果
- 同步更新的文档
- 当前回档点
- 若属于中大型修改：对应的 `src/wangyou/xiugai/*.md` 文件路径

若缺少其中任一项，说明本次执行未达到标准流程。
