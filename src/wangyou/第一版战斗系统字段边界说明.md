# 第一版战斗系统字段边界说明

> 目标：为 `src/wangyou/schema.ts`
> 中第一版战斗系统相关字段提供稳定边界，明确哪些字段属于长期档案，哪些字段只应在战斗期存在，以及运行时命令和日志的读写约束。

---

## 1. 字段分层

第一版战斗系统统一分成两层：

- 长期字段：长期保留在 `stat_data` 中，跨战斗存在，用于角色成长、背包、规则与定义。
- 战斗期字段：仅在有战斗实例时存在或变化，主要位于 `战斗状态` 中，战斗结束后可清理。

### 1.1 长期字段

以下字段属于长期数据：

- `世界`
- `规则配置`
- `技能定义表`
- `被动定义表`
- `角色档案`
- `背包`
- `金币`
- `战斗归档`

这些字段的职责：

- `规则配置`：提供结算规则、资源规则、检定规则。
- `规则配置.敌方掉落表`：提供按敌方角色 ID 索引的正式掉落配置来源。
- `技能定义表` / `被动定义表`：提供引擎执行的白名单定义。
- `角色档案`：保存角色长期属性、经验、已解锁技能、基础资源。
- `背包`：保存长期持有的道具库存。
- `金币`：保存战斗胜利、任务等长期货币结余。
- `战斗归档`：保存每场战斗的胜负、轮数、奖励摘要与时间戳，供后续战报或历史检索使用。

这些字段不应被 AI 直接按结果篡改。运行时变化必须先经脚本结算，再由脚本回写。

### 1.2 战斗期字段

以下字段属于战斗期运行时结构：

- `战斗状态.battleId`
- `战斗状态.状态`
- `战斗状态.模式`
- `战斗状态.回合数`
- `战斗状态.当前行动单位Id`
- `战斗状态.当前阶段`
- `战斗状态.行动游标`
- `战斗状态.参战方`
- `战斗状态.先攻队列`
- `战斗状态.日志`
- `战斗状态.随机种子`
- `战斗状态.玩家输入态`
- `战斗状态.待处理指令`
- `战斗状态.结算结果`

这些字段只描述当前战斗实例，不应被拿来作为长期角色档案的唯一来源。

## 2. 运行时结构常量说明

`schema.ts` 已拆出以下运行时 Schema 常量，作为第一版正式基线：

- `RuntimeUnitResourceSchema`
- `RuntimeUnitAttributesSchema`
- `RuntimeUnitFlagsSchema`
- `RuntimeSkillSchema`
- `RuntimeStatusSchema`
- `RuntimeModifierSchema`
- `RuntimeUnitSchema`
- `BattleSideRuntimeSchema`
- `BattleLogEntrySchema`
- `BattleLogPayloadSchema`
- `PendingCommandSchema`

约束如下：

- 运行时结构只服务于战斗引擎、状态栏和脚本桥接层。
- 长期档案字段不能直接复用成运行时单位实例，必须经过派生与初始化。
- 新增运行时字段时，优先扩展这些常量，再回接 `战斗状态` 主结构。

---

## 3. 待处理指令边界

`战斗状态.待处理指令` 使用 `PendingCommandSchema`，用于承载“已提交但尚未结算”的动作意图。

### 3.1 可写入方

允许写入者只有两类：

- 前端玩家操作：`clientHint.source = 'player_ui'`
- 敌方脚本 AI：`clientHint.source = 'enemy_ai'`

世界书文本、角色卡文本、AI 叙述层都不应直接写入最终数值结果，只能写动作意图。

### 3.2 消费方

消费方只有战斗引擎流程层：

- `engine.ts` 或后续桥接后的统一控制入口读取 `待处理指令`
- `commands.ts` 负责命令校验、补齐与兜底
- 通过校验后才进入效果执行与结算

### 3.3 清空时机

`待处理指令` 必须在以下时机清空：

- 当前指令执行完成后
- 当前指令被判定为非法后
- 战斗被重建、重置或清理时
- 战斗结束进入最终结算后

不允许让旧指令跨回合残留并被重复消费。

## 4. 日志 payload 最小结构

`战斗状态.日志[].payload` 使用 `BattleLogPayloadSchema`，第一版只约束最小白名单键：

- `roll`
- `damageBreakdown`
- `statusMeta`

说明如下：

### 4.1 `roll`

用于记录一次检定或掷骰摘要，建议包含：

- `rollType`
- `actorId`
- `targetId`
- `dice`
- `rawRolls`
- `finalRoll`
- `modifier`
- `dc` 或 `targetAC`
- `success`
- `isCriticalSuccess`
- `isCriticalFail`

### 4.2 `damageBreakdown`

用于记录伤害或治疗的关键拆分，建议包含：

- `base`
- `reduced`
- `shieldAbsorbed`
- `final`
- `damageType`

### 4.3 `statusMeta`

用于记录状态类效果附加或移除时的元信息，建议包含：

- `statusId`
- `duration`
- `stacks`
- `chance`
- `applied`

第一版允许在上述键之外继续增加额外字段，但必须保持对象结构，避免将不可解析文本直接塞入 `payload`。

## 5. 技能与道具字段说明

### 5.1 技能 `目标类型`

`技能定义表[*].目标类型` 当前可用值：

- `self`
- `single_enemy`
- `single_ally`
- `all_enemies`
- `all_allies`
- `random_enemy`

约定：

- 前端目标面板只应展示与该目标类型一致的目标。
- 引擎在没有显式目标时，可以按目标类型做最小兜底选择。
- 后续扩展多目标或复杂站位时，应优先扩展目标类型，不直接改写现有语义。

### 5.2 道具 `目标类型`

`背包[*].目标类型` 已纳入 Schema，第一版默认值为 `self`。

约定：

- 当前未接入完整道具目标面板前，可优先按 `self` 处理。
- 后续 `ItemPanel` 与 `TargetPanel` 联动时，应以该字段决定道具目标选择方式。
- 治疗药、法力药等第一版道具默认允许自用；需要指定友方目标时再显式改为 `single_ally`。

### 5.3 道具 `战斗可用`

`背包[*].战斗可用` 表示该道具是否允许出现在战斗期道具面板与命令校验结果中。

约定：

- `true`：可在战斗中被提交为 `item` 行动。
- `false`：只允许在战斗外使用，不应出现在战斗可用列表中。
- `commands.ts` 落地后，道具命令合法性必须检查该字段。

### 5.4 `规则配置.敌方掉落表`

第一版已支持正式掉落配置入口：`规则配置.敌方掉落表`。

结构如下：

```ts
敌方掉落表: Record<
  string,
  Array<{
    itemId: string;
    chance: number;
    minCount: number;
    maxCount: number;
  }>
>;
```

约定：

- key 使用敌方 `sourceCharacterId`，例如 `goblin_scout`。
- `settlement.ts` 在生成掉落时优先读取该字段。
- 若当前敌人没有配置正式掉落表，脚本才回退到“从现有可用背包物品池中推导”的最小可用实现。
- 后续内容化时应优先扩充该字段，而不是继续增强背包池推导逻辑。

---

## 6. 第一版维护原则

- `schema.ts` 负责定义边界，不负责执行逻辑。
- 引擎负责结算，前端负责收集意图，AI 只负责提交意图。
- 运行时单位结构可以演进，但必须维持向下兼容的最小字段集合。
- 第一版新增字段时，优先补文档，再补脚本与 UI 读写。

---

## 7. `BattleResultSummary` 使用说明

`战斗状态.结算结果` 是第一版战斗结束后的统一摘要对象，职责是承接“本场结算结论”，再由 `settlement.ts` 回写到长期字段。

当前字段语义如下：

- `winner`：胜负结果，取值为 `ally` / `enemy` / `draw` / `escape`。
- `rounds`：本场战斗总回合数。
- `expGain`：本场战斗为我方角色结算的总经验值；失败、平局、逃跑默认可为 `0`。
- `goldGain`：本场战斗结算的金币收益；默认只在我方胜利时产出。
- `summary`：面向前端结果面板和归档摘要的结论文本。
- `rewardTexts`：奖励文本数组，按“经验 / 金币 / 掉落”整理后的可展示字符串。
- `itemDrops`：脚本计算得到的掉落列表，采用 `{ itemId, count }[]` 结构。
- `characterChanges`：逐角色战后资源与成长变化快照。

约定如下：

- `BattleResultSummary` 只描述本场结算结果，不直接替代长期档案。
- 长期回写必须通过 `settlement.ts` 统一执行，不能由 UI 或 AI 直接写入 `角色档案`、`背包`、`金币`、`战斗归档`。
- `rewardTexts` 用于前端展示与归档摘要复用，避免 UI 额外拼接奖励文案。
- `characterChanges.newLevel` 允许在生成摘要时先做预测，在实际回写后再作为最终结果保留。
- `summary` 与 `rewardTexts` 当前已直接被结果面板消费，前端无需再二次拼接胜负摘要与奖励文本。

## 8. 战斗归档结构说明

`战斗归档` 属于长期字段，第一版采用以下稳定结构：

- `battleId`：战斗实例 ID。
- `winner`：胜负结果。
- `rounds`：总回合数。
- `summary`：本场结论摘要。
- `rewards`：奖励文本数组，通常直接来自 `BattleResultSummary.rewardTexts`。
- `timestamp`：归档写入时的世界时间。

约定如下：

- 每场战斗在 `applyBattleSummaryToRecords` 完成长期回写时追加一条归档。
- 归档只记录摘要，不保存完整运行时单位快照，避免长期数据膨胀。
- 需要详细战报时，应继续读取战斗内日志或后续新增专门战报结构，而不是把 `战斗状态` 直接长期保留。
