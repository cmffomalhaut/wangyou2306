## 对现有战斗系统的理解

### 1. 系统结构

- `index.ts` 负责从 MVU 变量提取队伍/技能/背包数据，启动 iframe + Vue 战斗界面，并在战斗结束后回写 HP、MP、金币、经验与背包消耗。
- `store.ts` 负责前端状态编排，连接 UI 与 `BattleEngine`，控制阶段流转：道具选择 -> 指令选择 -> 动画/结算 -> 强制换人/结果。
- `engine.ts` 是核心规则层，负责行动顺序、伤害、状态、换人、逃跑、捕捉、AI 决策、奖励结算。
- `types.ts` 定义了技能 DSL、状态效果、日志、战斗结果、敌方 AI 配置等战斗协议。

### 2. 核心流程

建议文档按下列顺序描述真实流程：

1. 从 `战斗控制台` 和 `角色数据` 组装 `BattleUnit`
2. 读取背包中的战斗道具
3. 初始化 `BattleEngine`
4. 敌方预先 `planEnemyAction`
5. 每回合开始时结算开场状态提示（冰冻/麻痹/混乱）
6. 处理换人/逃跑/技能释放
7. 按速度决定先手，并可能触发速度追加行动
8. 回合末结算 DoT、护盾同步、重盾/均衡型回合恢复、场地 buff 递减
9. 判定胜负、捕捉或战后回写

### 3. 已确认的重要公式与规则

文档中应明确区分“设计意图”和“当前真实实现”。

#### 属性克制

- 克制链：火克风、风克地、地克水、水克火、光克暗、暗克光
- 克制倍率：`1.5`
- 被克制倍率：`0.67`
- 同属性相撞：`0.5`
- 同属性加成（STAB）：技能元素与攻击者元素相同则 `1.35`

#### 暴击与随机

- 基础暴击率：`0.0625`
- 暴击倍率：`1.5`
- 总暴击率上限：`0.5`
- 伤害随机浮动：`0.92 ~ 1.0`

#### 命中公式

- 基础命中：`clamp(skill.命中率 or 0.95, 0.35, 1.0)`
- 命中/闪避修正：
  - `acc = clamp(1 + 命中增减总和, 0.6, 1.8)`
  - `eva = clamp(1 + 闪避增减总和, 0.6, 1.8)`
- 最终命中率：`clamp(base * (acc / eva), 0.3, 1.0)`

#### 基础伤害公式

- 当技能带 `系数` 时：
  - `rawDmg = atk * 系数 + 基础威力`
  - `defMitigation = def / (def + atk * 0.5)`
  - `base = rawDmg * (1 - defMitigation * 0.6) * 0.55`
- 当技能没有 `系数` 时，走兜底公式：
  - `ratio = (atk / def) ^ 0.92`
  - `base = (((2 * level) / 5 + 2) * basePower * ratio) / 60 + 2`
- 最终伤害主体：
  - `floor(base * 属性倍率 * STAB * 暴击 * 随机 * 1.15 * 增伤倍率)`

#### 伤害上限封顶

- 当前实现对单次伤害做了按目标 `HPMax` 比例封顶。
- 系数技能封顶更高，普通兜底技能封顶更低。
- 不同克制结果下封顶区间约为：
  - 系数技能：`13% ~ 38% HPMax`
  - 非系数技能：`12% ~ 28% HPMax`

#### 治疗 / 吸血 / 护盾

- 治疗：优先使用 `治疗量`，否则按 `目标HPMax * 治疗比例`
- 吸血：回复 `floor(造成伤害 * 吸血比例)`，默认比例下限上限会被钳制到 `0.1 ~ 1.0`
- 护盾：优先 `护盾值 + HPMax * 护盾比例`，并受 `shieldMax` 或 `HPMax` 上限约束

#### 状态效果

- 灼烧/中毒：回合末固定按 `HPMax * value` 扣血
- 冰冻：无法行动
- 麻痹：按 `value` 作为跳过行动概率，默认 `0.25`
- 混乱：按 `value` 作为自伤概率，默认 `0.35`；自伤伤害为 `6% HPMax`
- buff/debuff：直接以乘法或加减法作用于属性/命中/闪避

#### 速度追加行动

- 若攻击者速度高于防守者：
  - 基础追加概率 `min((速度差)/200, 0.35)`
  - `敏捷型`吃满概率，其他类型减半

#### 逃跑公式

- BOSS 战不可逃跑
- 普通逃跑成功率：`min(0.95, 0.3 + 0.1 * 尝试次数 + (己方速度 - 敌方速度) / 220)`
- 失败后敌方立即反击一次，并推进回合末结算

#### 捕捉公式

- 基础率按品质：`E 90% / D 75% / C 50% / B 25% / A 10% / S 1%`
- S 级硬锁：只有骰子 `=1` 才成功，其他修正全部失效
- 球倍率：普通 `1` / 高级 `1.5` / 超级 `2` / 魅惑 `1.2`
- 辅助器：`+15%`
- HP 压低修正：`<5% +20%`，`<=20% +10%`
- 状态修正：麻痹/冰冻/混乱存在时 `+15%`
- 意志抵抗：坚定 `-30%`，动摇 `-10%`
- 训练师技能加成：直接加到最终概率
- 最终：`clamp(baseRate * ballMultiplier + techMod + statusMod + resistMod + trainerBonus, 0, 1)`

### 4. 当前实现中的实际情况与偏差

文档需要明确写出这些“看规则不一定能看出来”的实际行为：

- `getAvailableSkills()` 用 `s.消耗MP` 判断可用，但真正扣蓝在 `executeSkill()` 中按稀有度固定值扣除，和 `技能原则.yaml` 一致，但与 `normalizePlayerCommand()` 中的 MP 校验存在口径不一致风险。
- `executeSkill()` 扣蓝依据稀有度，而不是技能字段 `消耗MP`；因此文档应说明“`消耗MP` 更像展示字段，真实战斗按稀有度扣除”。
- `executeRound()` 中玩家若选择逃跑，会直接调用 `tryEscape('普通')`，没有使用真实 `battleType`；这意味着捕获/BOSS 战的逃跑限制在这里可能表现失真，应在文档中标注为当前实现问题。
- `all_enemies` 目标类型在当前逻辑里并未真正遍历敌方全队，本质仍只打前排。
- `ally` 目标在当前 1v1 前排结构下等价于 `self`。
- `all_allies + buff` 被实现为“前排继承型场地 buff”，不是直接给全队挂独立状态。
- `all_allies + heal` 会治疗前排与后排。
- `buildStatusMessage()` 对 buff/debuff 的施加对象取决于 `executeSkill()` 传入的 defender；若技能目标是 self/ally，就是给自己加；若是 enemy，就是给对手上减益。
- 护盾先吸收伤害，再从剩余伤害中扣 HP，但战斗日志展示会写“总伤害 = 本体伤害 + 护盾吸收”。
- `强攻型/重盾型/魔导型/均衡型/敏捷型` 已有明确脚本特性，且与 `技能原则.yaml` 对齐。
- 单次行动结束后会递减冷却；若触发速度追加行动，同一技能会再次执行，并再次扣蓝、再次进 CD 递减链。

### 5. 重点优化方向

文档末尾建议加入“优化建议”，分为规则优化和代码优化两部分。

#### 规则层优化

- 统一 MP 规则：技能可用性判断、UI 展示、玩家指令校验、实际扣蓝应全部基于同一来源，避免 `消耗MP` 字段与稀有度固定扣蓝并存造成误解。
- 明确 `all_enemies` 语义：要么真正实现群攻，要么在 DSL 与文档中明确“当前版本仅前排结算”。
- 明确 `all_allies + buff` 是否应只影响前排继承，还是扩展为全队独立 buff。
- 重新评估伤害封顶：当前按 HPMax 比例硬封顶能稳回合数，但会压缩高爆发技能差异，建议列为平衡性优化项。
- 区分“控制状态提示”和“控制状态阻止行动”的责任，目前冰冻/麻痹/混乱存在提示与真实阻断分散在多个位置的问题。

#### 工程层优化

- 抽离统一公式模块：把命中、伤害、治疗、捕捉、逃跑等公式整理成独立章节/函数，减少 `engine.ts` 的巨型文件复杂度。
- 统一常量配置：暴击率、倍率、封顶、职业特性、品质捕捉率等应集中管理，便于策划调参。
- 建立“规则注释表”或参数表格，避免代码中的中文 DSL 字段与真实结算逻辑脱节。
- 为关键路径补测试：至少覆盖伤害公式、追加行动、all_allies heal/buff 差异、逃跑限制、捕捉硬锁、强制换人。
- 修复 `tryEscape('普通')` 的战斗类型硬编码问题。
